# AR UI / Detail UI Split

이 문서는 현재 프로젝트에서 `AR 환경 UI` 와 `상세정보 UI` 를 역할별로 분리해 설계하기 위한 기준 문서다.

## Summary

- `AR 환경 UI` 는 `uGUI(Canvas) + TMP + 런타임 오버레이` 중심으로 유지한다.
- `상세정보 UI` 는 `AR 환경을 벗어난 별도 페이지` 에서 `UI Toolkit` 으로 구성한다.
- 목적은 `AR 화면은 탐색과 1차 확인에 집중`, `상세 페이지는 읽기와 행동에 집중` 하도록 역할을 나누는 것이다.

## Why Split

- AR 화면은 실시간으로 상태가 바뀌고, 사용자 시선과 카메라 방향에 즉시 반응해야 한다.
- 상세정보 화면은 정적인 정보 레이아웃, 카드 구성, 스크롤, 버튼 설계가 더 중요하다.
- 두 성격을 같은 화면에서 동시에 처리하면 터치 동선과 정보 위계가 쉽게 꼬인다.
- 현재 프로젝트에서 가장 큰 문제였던 `상단 라벨 터치 부담`, `AR 위 상세 오버레이의 복잡함` 을 줄일 수 있다.

## AR Environment UI

### Goal

- 사용자가 주변 건물을 빠르게 탐색하고 선택하게 한다.
- AR 화면은 가능한 한 가볍고 즉시 반응해야 한다.
- 사용자는 휴대폰을 계속 중앙에 고정하지 않아도 간단한 정보를 읽을 수 있어야 한다.

### Recommended Tech

- `uGUI(Canvas)`
- `TextMeshPro`
- 현재 `ARUIManager`, `GeospatialManager`, `BuildingMarker` 중심 구조 유지

### Elements To Keep In AR

- 상태 안내 문구
  - 예: `앵커가 표시되고 있습니다`, `주변 앵커를 둘러보세요`
- 2D 마커 / 핀
- 마커 하단 1줄 이름 라벨
- 중앙 포커스 기준 건물 선택 로직
- 하단 고정 퀵카드
- `상세 보기` 진입 버튼

### Recommended AR Layout

- 상단: 상태 안내 문구
- 중앙/공간: 마커 및 건물 인지 요소
- 마커 하단: 건물명 1줄
- 하단: 선택된 건물의 고정 퀵카드
  - 건물명
  - 거리
  - 필요 시 카테고리 또는 짧은 설명 1줄
  - `상세 보기`

### Recommended Selection Behavior

- 사용자가 특정 마커를 중앙 포커스로 맞추면 그 건물이 현재 선택 대상이 된다.
- 마커 하단 1줄 라벨은 현재 보고 있는 건물 이름을 빠르게 확인하는 용도다.
- 하단 퀵카드는 `마지막으로 선택한 건물` 기준으로 유지한다.
- 사용자가 휴대폰을 아래로 내리거나 시선을 잠깐 바꿔도, 하단 퀵카드는 다음 마커를 명확히 선택하기 전까지 유지한다.
- 사용자가 다른 마커를 중앙에 맞추면 하단 퀵카드는 즉시 새 건물 정보로 교체된다.
- 따라서 `휙휙 지나가며 이름만 확인하는 사용자` 와 `하단 카드에서 천천히 읽는 사용자` 를 동시에 수용할 수 있다.

### AR UI Principles

- 한 손으로 조작 가능해야 한다.
- 상단 터치보다 하단 터치를 우선한다.
- AR 위에는 긴 텍스트와 많은 카드형 정보를 두지 않는다.
- 즉시 반응, 시인성, 선택 확신이 중요하다.
- 이름 확인용 정보와 읽기용 정보를 분리한다.

## Detail Information UI

### Goal

- 사용자가 선택한 건물의 정보를 읽고, 바로 행동하게 한다.
- AR 문맥과 분리된 안정적인 읽기 환경을 제공한다.

### Recommended Tech

- `UI Toolkit`
- `UIDocument + UXML + USS + Controller Script`

### Recommended Transition

- AR 화면 하단 퀵카드의 `상세 보기` 버튼 선택
- AR 화면에서 별도 상세 페이지로 전환
- 뒤로 가기 시 다시 AR 화면 복귀

### Minimum Detail Elements

- 헤더
  - 카테고리
  - 건물명
- 본문
  - 주소
  - 영업시간
  - 전화번호
- 액션
  - 전화
  - 공유
  - 카카오맵 이동

### Detail Page Principles

- 스크롤 없이 주요 정보가 1차로 들어오게 구성한다.
- 정보 카드와 액션 버튼은 앱형 레이아웃으로 정돈한다.
- AR 오버레이처럼 보이게 하지 않고, 독립된 정보 페이지처럼 설계한다.
- 버튼은 `읽기` 보다 `행동` 을 유도하는 역할이 명확해야 한다.

## Final Role Split

### uGUI In AR

- 상태 안내 문구
- 앵커 안내
- 건물 탐색용 2D 마커
- 마커 하단 1줄 이름 라벨
- 하단 고정 퀵카드
- AR 내 선택 흐름 제어

### UI Toolkit In Detail Page

- 건물 상세 헤더
- 주소 / 영업시간 / 전화번호 카드
- 공유 / 전화 / 지도 이동 버튼
- 앱형 상세 레이아웃 전체

## Recommended User Flow

1. 사용자가 AR 화면에서 주변 건물을 탐색한다.
2. 각 마커 하단 1줄 라벨로 건물 이름을 빠르게 확인한다.
3. 중앙 포커스 또는 선택 기준에 맞는 건물이 감지되면 하단 퀵카드가 해당 건물 정보로 갱신된다.
4. 사용자는 휴대폰을 조금 내리거나 걸어가면서도 하단 퀵카드로 건물명과 거리 등을 천천히 확인할 수 있다.
5. 다른 마커를 선택하기 전까지 하단 퀵카드는 유지된다.
6. 사용자가 `상세 보기` 를 누른다.
7. AR 화면을 벗어난 상세 페이지가 열린다.
8. 주소 확인, 전화, 공유, 카카오맵 이동 등의 행동을 수행한다.
9. 뒤로 가기로 AR 화면으로 복귀한다.

## Expected Benefits

- AR 화면이 복잡해지지 않는다.
- 휙휙 지나가며 보는 사용자는 마커 하단 이름만 빠르게 확인할 수 있다.
- 하단 퀵카드 덕분에 휴대폰을 잠깐 내려도 간단 정보를 읽을 수 있다.
- 상세정보 읽기 가독성이 올라간다.
- 상단 터치 부담이 줄어든다.
- 유지보수 시 `AR UI` 와 `상세 페이지 UI` 를 독립적으로 다듬을 수 있다.
- `uGUI` 와 `UI Toolkit` 의 강점을 각각 맞는 곳에 쓸 수 있다.

## Notes

- AR 안에서의 상세 오버레이는 최소화한다.
- AR 안에서는 `이름 확인`, `간단 정보 확인`, `상세 진입` 까지만 담당한다.
- 상세 페이지는 `앱 화면` 처럼 생각하고 설계한다.
- 향후 확장 시에도 `AR 탐색` 과 `정보 소비` 는 분리된 상태를 유지하는 것이 좋다.

## Deferred Review

### Detail Page Rendering Strategy

- 현재 상세정보는 `AR 환경을 완전히 벗어난 별도 씬` 이 아니라, `같은 씬 안에서 AR 위를 전체화면 UI로 덮는 방식` 으로 구현되어 있다.
- 이 방식은 전환이 빠르고 AR 세션을 유지하기 쉽다는 장점이 있다.
- 반면 상세정보를 보는 동안에도 카메라와 AR 세션이 계속 살아 있을 수 있어, 배터리와 발열 면에서는 불리할 가능성이 있다.

### Need To Revisit Later

- 상세정보를 오래 보는 사용자가 많다면, `상세 페이지에서는 카메라를 쉬게 하는 구조` 가 더 적절할 수 있다.
- 이 경우 `AR 환경에서 벗어난 별도 상세 페이지 -> 닫기 -> AR 환경 복귀` 흐름으로 바꾸는 것을 검토한다.
- 다만 별도 페이지 전환 방식은 `AR 세션 재개`, `추적 복구`, `복귀 속도`, `사용자 체감 끊김` 을 함께 검토해야 한다.

### Decision Reminder

- 현재는 다른 UI 구조를 먼저 정리하고,
- 이후 아래 기준으로 `오버레이 유지` 와 `별도 상세 페이지 전환` 중 어떤 방식이 더 적합한지 다시 판단한다.
  - 배터리/발열
  - AR 복귀 속도
  - 사용자 체감 자연스러움
  - 구현 및 유지보수 비용
