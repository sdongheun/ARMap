# 개발 스레드 작업 지시서

> 이 문서는 `PROJECT_MASTER.md`를 바탕으로, 개발 스레드가 당장 수정 또는 추가해야 할 작업만 분리한 실행 문서다.  
> 방향성 판단은 `PROJECT_MASTER.md`를 따른다. 구현 스레드는 이 문서를 보고 코드를 수정한다.

## 1. 작업 원칙

- `PROJECT_MASTER.md`는 총괄 방향 문서다.
- 이 문서는 현재 스프린트의 실행 지시 문서다.
- 개발 스레드는 원칙적으로 이 문서의 작업을 우선 수행한다.
- `PROJECT_MASTER.md`의 방향과 충돌하는 구현은 하지 않는다.
- 작업 완료 후에는 변경 파일, 구현 내용, 확인 필요 사항을 보고한다.

## 2. 현재 제품 목표

- 사용자가 현재 위치에서 주변 건물을 AR로 탐색할 수 있어야 한다.
- 보이는 건물은 여러 개일 수 있어도, 정보 카드는 항상 1개만 떠야 한다.
- 비중앙 대상은 지도 앱 스타일의 마커 모양으로 보여야 한다.
- 중앙에 온 대상은 마커에서 라벨로 자연스럽게 변하는 애니메이션이 있어야 한다.
- 라벨 선택 시 AR 오버레이 상세 패널이 아니라 별도 전체 화면 상세 페이지로 이동해야 한다.
- 화면 정중앙에 위치한 건물만 요약 정보 대상으로 판단해야 한다.
- 정중앙이 아닌 시야 내 건물은 정보 카드 대신 마커로만 보여야 한다.
- 사용자가 마커를 화면 중앙으로 가져오면 해당 건물의 요약 정보가 나타나야 한다.
- 현재 보고 있는 건물 뒤에 있는 건물은 감지 대상에서 최대한 제외해야 한다.
- 1차 목표 달성 후 AR 길찾기 기능을 확장한다.

## 3. 현재 우선순위

1. 같은 전방 시야선상에서는 가장 가까운 건물만 감지
2. 마커를 지도형 마커 디자인으로 변경
3. 정중앙 대상은 마커에서 라벨로 전환
4. 라벨 선택 시 전체 화면 상세 페이지로 이동
5. 데이터 로드 및 재로드 안정화

## 4. 지금 바로 진행할 작업

### TASK-A1. 전방 시야 단일 후보 규칙 적용

- 목적: 사용자가 바라보는 방향에서 같은 시야선상에 여러 건물이 겹칠 경우, 가장 가까운 건물 하나만 감지 대상으로 남긴다.
- 요구사항:
  - 최종 정보 카드 대상은 항상 1개만 유지한다.
  - 전방 시야에 후보가 여러 개 있어도 정보 대상은 1개만 선택한다.
  - 화면상 거의 같은 방향에 있는 후보들은 같은 전방 시야 그룹으로 본다.
  - 같은 그룹 안에서는 가장 가까운 건물만 최종 후보로 남긴다.
  - 뒤쪽 건물은 마커가 보일 수는 있어도 감지 카드 대상에서는 제외한다.
  - 예시:
    - 사용자의 전방에 `GS`가 있고 그 뒤에 `투썸`이 있는 경우, `GS`만 감지되어야 한다.
    - `투썸`은 같은 방향선상의 후면 후보로 제거되어야 한다.
    - 사용자의 정면에 `카페`가 있고, 그 뒤 직선상에 `음식점`이 있으며, 좌우로 `편의점`, `다른 카페`가 실제로 보이는 경우:
      - 정보 카드는 정면의 `카페` 1개만 떠야 한다.
      - `음식점`은 후면 후보로 제거되어야 한다.
      - `편의점`, `다른 카페`는 보이더라도 카드 대상이 아니라 마커로만 남아야 한다.
- 구현 힌트:
  - `GeospatialManager`의 타겟 선택 로직을 우선 검토한다.
  - 현재 후보별 `viewport position`과 카메라 거리값을 함께 사용한다.
  - 화면 좌표가 일정 범위 안에서 겹치는 후보들을 같은 그룹으로 묶는다.
  - 그룹 내부에서는 가장 가까운 후보만 유지하고 나머지는 감지 후보에서 제거한다.
  - 완전한 차폐 판정이 아니라 정책 기반 제거를 먼저 적용한다.
- 완료 기준:
  - `GS`와 `투썸`처럼 같은 방향선상에 겹친 건물 중 가까운 것 하나만 감지된다.
  - 뒤에 있는 건물은 동시에 감지 카드 대상으로 뜨지 않는다.
  - 현재 프레임에서 감지 대상은 항상 1개만 유지된다.
  - 정면 카페, 좌우 편의점/다른 카페, 후면 음식점 사례에서 카드 대상은 정면 카페 1개만 유지된다.

### TASK-A2. 정중앙 건물만 정보 카드 연결

- 목적: 전방 시야 단일 후보 중에서도 화면 정중앙 조건을 만족하는 경우에만 정보 카드를 띄운다.
- 요구사항:
  - 화면 중앙 기준 허용 범위를 명시적으로 둔다.
  - 허용 범위를 벗어난 대상은 감지 상태로 전환하지 않는다.
  - 단일 후보가 존재해도 중앙 조건을 만족하지 않으면 정보 카드는 띄우지 않는다.
  - 중앙에 있지 않은 좌우 건물은 보여도 카드 대상이 아니라 마커 전용이어야 한다.
- 구현 힌트:
  - `FindBestTargetFromExistingAnchors()`와 UI 상태 전환을 같이 본다.
  - 필요하면 정중앙 판정 임계값을 인스펙터 조정값으로 분리한다.
- 완료 기준:
  - 화면 중앙 건물만 정보 카드 대상이 된다.
  - 중앙에 있지 않은 후보는 카드 없이 마커로만 남는다.

### TASK-A3. 비중앙 건물 마커 전용 처리

- 목적: 사용자가 보고 있는 방향의 주변 건물은 보조 마커로만 표시하고, 정보 카드는 정중앙 대상에만 연결한다.
- 요구사항:
  - 시야 안의 주변 건물은 계속 마커로 보인다.
  - 마커 디자인은 단순 점이 아니라 실제 지도 서비스에서 사용하는 핀 형태에 가까운 시각 언어를 사용한다.
  - 추천 방향:
    - 기본형은 핀 또는 드롭 형태
    - 선택 전에는 아이콘 또는 점 중심의 단순 형태
    - 선택 시 라벨형 카드로 확장 가능한 구조
  - 선택된 건물과 비선택 건물의 상태 차이가 명확해야 한다.
  - 비중앙 건물은 상세 정보 진입 트리거가 되면 안 된다.
- 구현 힌트:
  - `UpdatePreviewMarkers`, `UpdateScreenSpaceMarkers`, UI 상태 전환 흐름을 같이 본다.
  - 감지 대상과 미감지 대상의 UI 책임을 분리한다.
  - `BuildingMarker`의 시각 상태를 단순 색상/크기 변경이 아니라 `마커 상태`와 `라벨 상태`로 확장하는 방향을 검토한다.
- 완료 기준:
  - 비중앙 건물은 지도형 마커 디자인으로만 보인다.
  - 화면 중앙 건물만 라벨 또는 정보 카드 대상이 된다.
  - 나머지 건물은 마커로만 남는다.

### TASK-A4. 위치 이동 재조회 안정화

- 목적: 사용자가 이동하면 주변 데이터가 안정적으로 다시 로드되어야 한다.
- 요구사항:
  - 중복 재호출이 과도하게 발생하지 않아야 한다.
  - 재조회 중 UI와 앵커 상태가 불안정하게 깜빡이지 않아야 한다.
  - 재조회 직후 새 데이터 기준으로 감지가 정상 동작해야 한다.
- 구현 힌트:
  - 현재 위치 비교와 재호출 트리거 조건을 검토한다.
  - 재조회 중복 실행 방지 플래그가 필요할 수 있다.
- 완료 기준:
  - 일정 거리 이동 후 한 번의 안정적인 재조회가 수행된다.
  - 중복 로딩으로 인해 앵커가 반복 생성되지 않는다.

### TASK-A5. 예외 처리 및 사용자 안내 보강

- 목적: 위치 실패, API 실패, 데이터 없음 상황에서 앱이 조용히 실패하지 않도록 한다.
- 요구사항:
  - 위치 권한 거부
  - 위치 서비스 실패
  - API 호출 실패
  - 검색 결과 없음
  - 앵커 생성 실패
  - 위 상황에 대한 최소한의 로그 또는 사용자 안내가 필요하다.
- 완료 기준:
  - 치명적 실패 없이 앱이 상태를 유지한다.
  - 디버깅 가능한 로그가 남는다.

### TASK-A6. 마커 -> 라벨 전환 애니메이션 적용

- 목적: 사용자가 화면을 돌려 특정 마커를 중앙으로 가져오면, 해당 마커가 네모난 라벨 형태로 자연스럽게 변해야 한다.
- 요구사항:
  - 중앙이 아닌 상태에서는 지도형 마커만 보인다.
  - 중앙 판정이 되면 마커가 즉시 교체되는 것이 아니라 애니메이션으로 라벨로 확장된다.
  - 라벨은 과도한 정보 없이 빠르게 읽히는 수준으로 유지한다.
  - 추천 라벨 정보 구성:
    - 1줄: 건물명
    - 2줄: 카테고리 + 거리
    - 주소는 기본값에서 제외하고, 필요 시 한 줄 축약형으로만 검토한다.
  - 거리 표시는 `약 n m` 형태의 간단한 거리 정보로 제공한다.
- 구현 힌트:
  - 현재 오브젝트를 교체하는 방식보다, 하나의 마커 오브젝트가 상태에 따라 `핀 -> 라벨`로 변형되는 방식이 UI 일관성에 유리하다.
  - 애니메이션은 크기 확장, 배경 카드 확장, 텍스트 페이드 인 조합이 적절하다.
  - 사용자가 중앙에서 벗어나면 다시 라벨 -> 마커 상태로 복귀해야 한다.
- 완료 기준:
  - 비중앙 상태에서는 지도형 마커만 보인다.
  - 중앙 상태에서는 해당 마커가 라벨로 전환된다.
  - 라벨에는 `건물명 + 카테고리 + 거리`가 읽기 좋은 밀도로 표시된다.

### TASK-A7. 상세 정보 진입 흐름을 전체 화면 페이지로 전환

- 목적: 사용자가 라벨을 선택하면 AR 환경 위의 상세 오버레이가 아니라 별도 전체 화면 상세 페이지로 이동하게 한다.
- 요구사항:
  - 라벨 선택이 상세 진입의 유일한 명확한 트리거가 되어야 한다.
  - 상세 정보 진입 시 사용자는 AR 탐색 상태에서 분리되어 상세 페이지를 본다.
  - 상세 페이지에서는 건물명, 카테고리, 전체 주소, 전화, 지도 링크, 입점 시설 정보를 나열한다.
  - 상세 페이지에서 뒤로 가면 다시 AR 탐색 상태로 복귀할 수 있어야 한다.
- 구현 힌트:
  - 현재 `ARUIManager`의 상세 패널 구조는 재사용할 수 있는지 검토하되, 최종 UX는 오버레이가 아니라 전체 페이지 전환으로 본다.
  - 사실상 `AR 탐색 화면`과 `상세 정보 화면`을 분리된 상태로 관리해야 한다.
- 완료 기준:
  - 라벨 선택 시 전체 화면 상세 페이지로 전환된다.
  - 기존 AR 위 하단 시트형 상세 패널은 1차 목표에서 제거하거나 비활성화한다.
  - 상세 페이지에서 필요한 정보가 스크롤 가능한 형태로 정리된다.

### TASK-B1. 장소 검색 범위 확장

- 목적: 현재 음식점, 카페, 마트 등 일부 카테고리에 한정된 장소 검색 범위를 넓혀 더 다양한 건물/시설을 인식할 수 있게 한다.
- 배경:
  - 현재 `GeospatialManager`는 제한된 `categoryCodes`만 사용해 주변 장소를 가져오고 있다.
  - 이 방식은 상업시설 인식에는 유리하지만, 대학 건물이나 원룸 같은 비상업 건물 인식에는 한계가 있다.
  - 김해시 인제대학교 기준으로 `대학교 건물`, `원룸`, `술집`, `기숙사`, `학교부속시설` 등도 탐색 대상에 포함하고 싶다.
- 요구사항:
  - 기존 카테고리 검색 범위를 확장 가능 구조로 바꾼다.
  - 일반 상권용 카테고리와 캠퍼스용 카테고리 구성을 분리 가능하게 한다.
  - 카테고리 검색만으로 부족한 대상은 키워드 검색을 추가로 수행할 수 있게 구조를 확장한다.
  - 카테고리 검색 결과와 키워드 검색 결과는 하나의 후보 리스트로 합친다.
  - 중복 장소는 이름, 주소, 좌표 기준으로 정리한다.
  - 검색 결과가 많아질 경우 우선순위 규칙을 둘 수 있게 설계한다.
- 구현 방향:
  - 1차:
    - 현재 `categoryCodes`를 확장 가능한 설정값으로 정리한다.
    - 기존 카테고리 외에 술집, 숙박, 공공기관, 문화시설, 학교 관련 카테고리 확장을 검토한다.
    - 실제 카카오 카테고리 코드는 개발 스레드에서 공식 문서 기준으로 재확인한다.
  - 2차:
    - 카카오 `keyword search` API를 병행할 수 있게 구조를 추가한다.
    - 캠퍼스 건물처럼 카테고리 검색만으로 잘 안 잡히는 대상은 키워드 검색으로 보강한다.
  - 3차:
    - 캠퍼스 안내가 핵심 기능일 경우, 장기적으로 학교 건물 전용 데이터셋 도입 가능성을 열어둔다.
- 주의사항:
  - `원룸`은 카카오 카테고리 기반만으로는 안정적으로 인식되지 않을 수 있다.
  - `대학교 건물`은 지도 앱에서 보이더라도 카테고리 검색만으로 누락될 수 있다.
  - 검색 대상이 늘어나면 현재 클러스터링 방식이 잘못된 대표 건물명을 만들 수 있으므로, 검색 확장과 함께 병합/정규화 규칙도 같이 검토해야 한다.
- 완료 기준:
  - 기존 상업시설 외에 더 다양한 건물/시설 후보가 실제로 수집된다.
  - 인제대학교 주변에서 대학 건물 관련 후보가 기존보다 더 많이 잡힌다.
  - 술집, 숙박, 학교부속시설 등 확장 대상이 검색 결과에 포함된다.
  - 검색 결과가 늘어나도 중복이 과도하게 쌓이지 않는다.

### TASK-B2. 인제대학교 기준 키워드 검색 보강

- 목적: 김해시 인제대학교 주변에서 카테고리 검색만으로 누락되는 학교 건물과 생활권 시설을 키워드 검색으로 보강한다.
- 요구사항:
  - 인제대학교 기준 검증용 키워드 세트를 정의한다.
  - 키워드 검색 결과를 카테고리 검색 결과와 병합할 수 있어야 한다.
  - 키워드 기반으로 잡힌 캠퍼스 건물은 일반 상업시설과 구분 가능한 형태로 유지할 수 있게 한다.
- 추천 키워드 예시:
  - `인제대학교`
  - `인제대학교 본관`
  - `인제대학교 도서관`
  - `인제대학교 학생회관`
  - `인제대학교 기숙사`
  - `인제대학교 OO관`
  - `인제대 원룸`
- 구현 힌트:
  - 현재 `search/category` 흐름과 별도로 `search/keyword` 흐름을 추가하는 구조를 검토한다.
  - 키워드 결과와 카테고리 결과를 합칠 때 중복 제거 규칙이 먼저 필요하다.
  - 키워드 리스트는 하드코딩보다 설정값으로 분리하는 쪽이 낫다.
- 완료 기준:
  - 카테고리 검색에서 잘 안 잡히던 인제대학교 건물 후보가 키워드 검색으로 보강된다.
  - 키워드 검색이 추가돼도 동일 장소가 과도하게 중복 표시되지 않는다.

### TASK-B3. 인제대학교 기준 데이터 수집 검증

- 목적: 인제대학교 주변에서 실제로 무엇이 카테고리 검색으로 가능하고, 무엇이 키워드 검색 또는 별도 데이터셋이 필요한지 검증한다.
- 요구사항:
  - 현재 카테고리 검색 결과 로그 수집
  - 확장 카테고리 검색 결과 비교
  - 키워드 검색 결과 비교
  - `인제대학교 OO관`, `기숙사`, `원룸`, `술집` 인식 여부 확인
- 검증 결과로 남겨야 할 것:
  - 카테고리 검색만으로 가능한 대상
  - 키워드 검색이 필요한 대상
  - 별도 데이터셋이 필요한 대상
- 완료 기준:
  - 인제대학교 기준 데이터 수집 한계와 보강 방식이 구분되어 정리된다.
  - 개발 스레드가 이후 어떤 수집 전략으로 갈지 판단할 수 있는 검증 결과를 남긴다.

### TASK-C1. 한 건물 내 복수 시설 처리 정책 적용

- 목적: 하나의 건물 또는 동일 위치 그룹 안에 여러 시설이 존재할 경우, AR 화면에서는 정보 과밀을 막고 상세 페이지에서 시설 탐색이 가능하도록 역할을 분리한다.
- 정책:
  - AR 화면에서는 대표 건물 또는 대표 위치 1개만 라벨로 보여준다.
  - 같은 건물에 속한 여러 시설을 AR에서 각각 정보 카드로 띄우지 않는다.
  - 같은 건물 안의 다른 시설들은 상세 페이지에서 리스트 형태로 제공한다.
- 배경:
  - 학교 부속시설이나 상가 건물처럼 하나의 건물 안에 여러 장소가 붙는 경우가 늘고 있다.
  - 이들을 AR에서 모두 정보 대상으로 처리하면 다시 정보 과밀 상태가 된다.
  - 현재 프로젝트의 핵심은 `AR에서는 빠른 건물 식별`, `상세에서는 시설 탐색`으로 역할을 나누는 것이다.
- 요구사항:
  - 하나의 건물/그룹에는 대표 라벨 1개만 유지한다.
  - 상세 페이지에는 그 건물에 속한 시설 목록을 별도 섹션으로 보여준다.
  - 시설 리스트는 이름, 카테고리, 전화번호, 지도 링크 등 현재 보유한 정보를 활용한다.
  - 대표 라벨은 개별 매장명이 아니라, 가능하면 건물명 또는 대표 위치명을 우선 사용한다.
- 구현 힌트:
  - 현재 `BuildingData`와 `FacilityInfo` 구조를 활용해 대표 정보와 하위 시설 목록을 분리한다.
  - 클러스터링 결과를 하나의 `BuildingData`로 유지하되, 내부 시설 목록을 상세 페이지에 전달하는 흐름을 명확히 한다.
  - AR 탐색 단계에서는 `facilities`를 직접 노출하지 않고, 대표 라벨만 보여주는 방향으로 UI 책임을 분리한다.
- 완료 기준:
  - 한 건물에 여러 시설이 있어도 AR 화면에서는 대표 라벨 1개만 보인다.
  - 상세 페이지에서는 해당 건물의 시설 리스트를 확인할 수 있다.
  - 복수 시설이 있는 건물에서도 AR 정보 과밀이 발생하지 않는다.

### TASK-C2. 대표 라벨 선정 규칙 정리

- 목적: 한 건물에 여러 시설이 있을 때 어떤 이름을 AR 대표 라벨로 보여줄지 일관된 규칙을 둔다.
- 요구사항:
  - 대표 라벨 선정 규칙을 코드와 문서에서 일관되게 유지한다.
  - 단순히 첫 번째 검색 결과를 대표 이름으로 쓰지 않는다.
  - 가능하면 건물명 또는 공식 시설명을 우선한다.
- 추천 우선순위:
  - 1순위: 건물명 또는 학교/공공기관의 공식 명칭
  - 2순위: 사용자가 가장 쉽게 인지할 수 있는 대표 시설명
  - 3순위: 동일 그룹 내 중심 좌표에 가장 가까운 시설 또는 정렬 우선순위가 높은 시설
- 주의사항:
  - 대표 라벨이 특정 입점 매장명으로 고정되면 사용자가 건물 전체를 오해할 수 있다.
  - 캠퍼스 건물은 개별 매장보다 `OO관`, `기숙사`, `학생회관` 같은 건물 단위 명칭이 우선되어야 한다.
- 완료 기준:
  - 대표 라벨이 매번 흔들리지 않고 일관된 기준으로 선택된다.
  - 캠퍼스 건물과 상가 건물 모두에서 사용자 입장에서 이해 가능한 이름이 노출된다.

## 7. PM 검토 필요 항목

- "건물"의 정의를 실제 건물 단위로 고정할지, 대표 시설 단위까지 허용할지
- 정중앙 판정 허용 범위를 얼마나 엄격하게 둘지
- 후면 건물 제거를 정책 기반으로 먼저 처리할지, 더 정교한 판정으로 갈지
- 라벨에 축약 주소를 기본 포함할지, `건물명 + 카테고리 + 거리`만 기본으로 둘지

## 8. 필수 테스트 케이스

- 테스트 케이스 1:
  - 상황: 전방에 `GS`, 그 뒤에 `투썸`
  - 기대 결과: 정보 카드는 `GS`만 표시, `투썸`은 카드 대상 제외
- 테스트 케이스 2:
  - 상황: 정면에 `카페`, 그 뒤 직선상 `음식점`, 좌우에 `편의점`, `다른 카페`
  - 기대 결과: 정보 카드는 정면 `카페` 1개만 표시
  - 기대 결과: `음식점`은 후면 후보 제거
  - 기대 결과: `편의점`, `다른 카페`는 마커만 표시
- 테스트 케이스 3:
  - 상황: 비중앙 상태의 건물이 화면 중앙으로 이동
  - 기대 결과: 지도형 마커가 라벨로 애니메이션 전환
  - 기대 결과: 라벨에는 `건물명 + 카테고리 + 거리` 표시
- 테스트 케이스 4:
  - 상황: 중앙 라벨을 사용자가 선택
  - 기대 결과: AR 오버레이가 아닌 전체 화면 상세 페이지로 전환
  - 기대 결과: 뒤로 가면 다시 AR 탐색 상태 복귀
- 테스트 케이스 5:
  - 상황: 인제대학교 주변에서 기존 카테고리 검색과 확장 카테고리 검색을 비교
  - 기대 결과: 기존보다 더 다양한 건물/시설 후보가 수집됨
- 테스트 케이스 6:
  - 상황: `인제대학교 본관`, `인제대학교 도서관`, `인제대학교 기숙사`, `인제대 원룸` 키워드 검색
  - 기대 결과: 카테고리 검색으로 누락되던 대상 일부가 보강됨
- 테스트 케이스 7:
  - 상황: 카테고리 검색 결과와 키워드 검색 결과를 병합
  - 기대 결과: 동일 대상이 과도하게 중복 표시되지 않음
- 테스트 케이스 8:
  - 상황: 하나의 건물 안에 카페, 편의점, 행정실 등 여러 시설이 함께 존재
  - 기대 결과: AR 화면에는 대표 라벨 1개만 표시
  - 기대 결과: 상세 페이지에는 시설 리스트가 정리되어 표시
- 테스트 케이스 9:
  - 상황: 학교 건물 또는 복합 상가에서 대표 라벨 선정
  - 기대 결과: 특정 입점 매장명보다 건물명 또는 대표 위치명이 우선 표시
