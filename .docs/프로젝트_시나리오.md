# 프로젝트 시나리오

## 목적

이 문서는 현재 프로젝트 코드 기준으로, 사용자가 앱을 실행한 뒤 휴대폰 카메라로 실제 건물을 바라보며 정보를 확인하는 과정이 개발자 관점에서 어떻게 처리되는지 최종 정리한 문서다.

현재 시스템의 핵심 전제는 다음과 같다.

- 이 앱은 카메라 영상에서 건물 외곽을 직접 비전 모델로 인식하지 않는다.
- 현재 구현은 `GPS + ARCore Geospatial Pose + Heading + 카메라 전방 벡터 + Kakao 장소 데이터`를 조합해 사용자가 보고 있을 가능성이 높은 건물을 선택한다.
- 즉 "건물 인식"은 이미지 기반 객체 인식이 아니라, 주변 건물 후보 중에서 현재 위치와 방향에 가장 맞는 대상을 선택하는 방식이다.

관련 핵심 스크립트는 다음과 같다.

- `ARmap_Pro_Max_Ultra_Plus/Assets/Scripts/Geospatial/GeospatialManager.cs`
- `ARmap_Pro_Max_Ultra_Plus/Assets/Scripts/Geospatial/GeospatialManager.Search.Flow.cs`
- `ARmap_Pro_Max_Ultra_Plus/Assets/Scripts/Geospatial/GeospatialManager.Clustering.Flow.cs`
- `ARmap_Pro_Max_Ultra_Plus/Assets/Scripts/Geospatial/GeospatialManager.Buildings.Factory.cs`
- `ARmap_Pro_Max_Ultra_Plus/Assets/Scripts/AnchorVisibilityPlanner.cs`
- `ARmap_Pro_Max_Ultra_Plus/Assets/Scripts/AnchorPoolPlanner.cs`
- `ARmap_Pro_Max_Ultra_Plus/Assets/Scripts/BuildingMarker.cs`
- `ARmap_Pro_Max_Ultra_Plus/Assets/Scripts/ARUI/ARUIManager.Cards.cs`
- `ARmap_Pro_Max_Ultra_Plus/Assets/Scripts/ARUI/ARUIManager.Detail.cs`

## 1. 앱 실행 직후

앱이 시작되면 `GeospatialManager.Start()`가 실행되고, 아래 순서로 초기화가 진행된다.

1. `LocalApiKeys.json`에서 Kakao API 키와 TMap 키를 읽는다.
2. `Camera.main.transform`을 캐시한다.
3. `ARUIManager` 이벤트를 연결한다.
4. 위치 권한을 확인하고 `Input.location.Start()`로 GPS를 시작한다.
5. 위치 서비스가 `Running`이 될 때까지 대기한다.
6. 현재 GPS 기준으로 `ReloadDataAtCurrentLocation()`를 호출해 주변 장소 데이터를 다시 수집한다.
7. `EarthManager.EarthTrackingState == TrackingState.Tracking`이 될 때까지 대기한다.

이 단계에서 사용자는 아직 건물을 바로 보는 것이 아니라, 앱이 사용할 위치/트래킹/주변 데이터 기반을 먼저 준비하게 된다.

## 2. 주변 장소 수집

건물 정보의 원재료는 카카오 로컬 API 응답이다. `FetchAndClusterNearbyPlaces()`에서 현재 좌표를 중심으로 주변 장소를 여러 방식으로 수집한다.

현재 검색 소스는 크게 세 가지다.

- 상권 카테고리 검색
- 캠퍼스 관련 카테고리 검색
- 키워드 검색

대표 키워드는 다음과 같은 형태다.

- `인제대학교`
- `인제대학교 본관`
- `인제대학교 도서관`
- `인제대 원룸`
- `인제대 술집`

검색 기본 반경은 `searchRadius = 200m`, 페이지는 `maxSearchPages = 3`까지 순회한다.

검색 결과는 `MergeFetchedPlaces()`에서 병합된다. 이때 동일 장소는 `정규화된 장소명 + 주소` 또는 `정규화된 장소명 + 좌표`를 기준으로 중복 제거된다. 여러 검색에서 같은 장소가 잡히면 더 품질이 좋은 문서를 남긴다.

즉 앱은 API 결과를 그대로 쓰지 않고, 하나의 정리된 장소 목록으로 정제한 뒤 다음 단계로 넘긴다.

## 3. 장소를 건물 단위로 묶기

사용자가 실제로 보게 되는 대상은 개별 상점 문서가 아니라 `BuildingData`다. 이 객체가 "하나의 건물"을 대표한다.

### 3-1. 주소 기준 1차 그룹화

`ClusterAndBuildList()`는 먼저 `GetBestAddress(document)`를 기준으로 같은 주소끼리 묶는다.

- 도로명 주소가 있으면 도로명 주소 사용
- 없으면 지번 주소 사용

즉 같은 주소를 공유하는 POI는 우선 같은 건물 후보로 본다.

### 3-2. 거리/토큰 기준 2차 분리

주소가 같다고 반드시 같은 건물은 아니므로, 다시 거리 기반 또는 기관형 토큰 기반으로 세분화한다.

- 일반 건물은 `clusterRadius = 20m` 기준 거리 클러스터링
- 학교/병원/관/동 같은 기관형 시설은 대표 토큰 기반으로 건물 그룹을 분리

### 3-3. 최종 BuildingData 생성

`ProcessClusterIntoBuildingData()`가 최종 `BuildingData`를 만든다.

- `buildingName`
- `latitude`, `longitude`
- `description`
- `lotNumberAddress`, `roadAddress`, `fetchedAddress`
- `phoneNumber`, `placeUrl`
- `facilities`

중요한 점은 건물 좌표가 단순 평균이 아니라, `SelectAnchorRepresentativeDocument()`가 고른 대표 문서 좌표를 그대로 사용한다는 것이다. 따라서 현재 앵커는 "건물 전체 중심"이 아니라 "그 건물을 대표하는 POI 한 점"에 생성된다.

## 4. 앵커 후보 목록 준비

데이터 로드가 끝나면 `CreateAllBuildingAnchors()`가 앵커 후보 목록을 준비한다.

현재 흐름은 다음과 같다.

1. 기존 앵커와 마커를 정리한다.
2. `GetBuildingsForAnchorCreation()`으로 앵커 후보 건물 목록을 만든다.
3. Geospatial Tracking이 안정될 때까지 대기한다.
4. 각 건물 altitude를 현재 카메라 고도 기준으로 맞춘다.
5. 이 단계에서는 모든 앵커를 한 번에 생성하지 않고, 후보 목록만 `_anchorCandidateBuildings`에 유지한다.

현재 코드 기준으로 `GetBuildingsForAnchorCreation()`은 더 이상 `anchorCreationRadius`나 `maxWorldTextMarkers`로 후보를 자르지 않는다. 즉 preview 점 후보는 거리 때문에 먼저 사라지지 않도록, 로드된 건물 목록 전체를 거리순으로 유지한다.

이 점이 예전 구조와의 가장 큰 차이다.

- 예전: 멀어지면 점 후보 자체가 잘림
- 현재: 점 후보는 거리로 자르지 않고 남겨둠

## 5. 실제 건물 선택 로직

사용자가 휴대폰으로 건물을 보기 시작하면 `Update()`에서 매 프레임 `CheckBuildingDetection()`가 실행된다.

### 5-1. 입력값

매 프레임 아래 값이 사용된다.

- 현재 Geospatial Pose
- 현재 Earth Tracking 상태
- 현재 Heading
- 카메라 전방 벡터
- `_anchorCandidateBuildings`

### 5-2. 1차 visibleCandidates

`AnchorVisibilityPlanner.CalculateVisibleCandidates()`가 먼저 "현재 방향상 보일 수 있는 건물"을 추린다.

현재 필터는 다음과 같다.

- Earth Tracking이 유효해야 한다.
- Heading Accuracy가 임계값보다 좋아야 한다.
- 현재 Heading과 건물 방위각 차이가 `detectionAngle / 2` 이하여야 한다.

중요한 점은 현재 preview 후보는 더 이상 거리로 자르지 않는다는 것이다. 즉 점이 사라지는 기준은 거리보다 방향, Heading 품질, 트래킹 상태다.

### 5-3. 화면상의 가상 X 계산

이 프로젝트는 실제 건물 메쉬를 카메라 영상 위에 투영해서 정밀하게 가시성을 계산하지 않는다. 대신 Heading 차이를 정규화해서 가상의 `viewportX`를 만든다.

- 정면이면 `0.5`
- 왼쪽이면 `0.5`보다 작아짐
- 오른쪽이면 `0.5`보다 커짐

즉 "이 건물이 화면 중앙에 얼마나 가까운가"를 방위각 차이로 근사한다.

### 5-4. previewCandidates와 topCandidates 분리

현재 구조에서는 후보가 두 층으로 나뉜다.

- `previewCandidates`
  - 방향상 보이는 건물 목록
  - preview 핀과 앵커 풀의 기준
- `topCandidates`
  - `detectionRadius` 안쪽 건물 중
  - 겹치는 후면 후보를 제거하고
  - 중심 우선으로 다시 정리한 목록
  - 최종 `focusedBuilding` 선택 기준

즉 현재 로직은 "점은 넓게, 텍스트는 더 엄격하게" 동작한다.

### 5-5. focusedBuilding 선택

`DetermineFocusedBuilding()`은 `topCandidates` 중에서 실제로 "지금 보고 있다"고 판단할 건물을 선택한다.

기준은 다음과 같다.

- 화면 중심 `(0.5, 0.5)`과의 거리
- 건물까지의 실제 거리
- 카메라 pitch가 `verticalAngleLimit = 60도`를 넘으면 선택 안 함
- `centerViewportThreshold` 안이면 우선 선택
- 안 되면 완화된 threshold에서 fallback 선택

즉 점 후보는 살아 있어도, 텍스트는 가까운 중심 건물 1개만 선택된다.

## 6. 세로/가로 모드 차이

세로모드와 가로모드는 같은 로직을 쓰되, 일부 탐지 파라미터가 달라진다.

현재 차이는 다음과 같다.

- 세로 `detectionRadius = 100`
- 가로 `landscapeDetectionRadius = 180`
- 세로 `detectionAngle = 130`
- 가로 `landscapeDetectionAngle = 180`
- 세로 `maxPreviewAnchors = 3`
- 가로 `landscapeMaxPreviewAnchors = 5`

즉 가로모드에서는 더 멀고 넓은 범위를 허용하고, 동시에 더 많은 preview 후보를 유지한다.

다만 현재도 "중앙 텍스트 1개"라는 기본 구조는 동일하다.

## 7. 앵커 생성과 유지

`RequestAnchorPoolForCandidates()`는 `previewCandidates`를 기준으로 실제 AR 앵커 풀을 맞춘다.

현재 흐름은 다음과 같다.

1. `previewCandidates`에서 중복 없는 `desiredBuildings`를 만든다.
2. 이전 프레임과 동일한 집합이면 아무것도 하지 않는다.
3. 달라졌으면 `SyncAnchorPool()`을 실행한다.
4. 필요 없는 앵커는 제거하고, 필요한 앵커만 새로 만든다.

앵커 자체는 다음 방식으로 생성된다.

```csharp
ARGeospatialAnchor anchor = AnchorManager.AddAnchor(
    building.latitude,
    building.longitude,
    building.altitude,
    Quaternion.identity);
```

현재 고도는 `building.altitude = camera altitude + markerAltitudeOffsetMeters`인데, 지금 `markerAltitudeOffsetMeters = 0.0`으로 맞춰져 있다. 즉 앵커 생성 기준 고도는 현재 휴대폰 카메라 고도와 동일하다.

## 8. 마커 생성과 배치

앵커가 생성되면 그 자식으로 `buildingMarkerPrefab`이 붙는다. 이때 사용자가 실제로 보는 것은 앵커 그 자체가 아니라, 앵커 아래에 붙은 `BuildingMarker`다.

### 8-1. 앵커는 건물 좌표, 마커는 카메라 쪽

현재 구조에서 중요한 점은 다음과 같다.

- 앵커는 건물 대표 POI 좌표에 생성된다.
- 하지만 실제 마커 시각물은 `ApplyMarkerPlacement()`에서 카메라 쪽으로 당겨진다.

현재 당김 로직은 대략 다음과 같다.

1. `anchorPosition`을 구한다.
2. `anchor -> camera` 방향 벡터를 구한다.
3. `cameraDistance * worldMarkerShellDistanceScale` 만큼 전진 거리를 만든다.
4. 이를 `worldMarkerShellMinRadius`, `worldMarkerShellMaxRadius`, `worldMarkerShellNearLimitFactor`로 제한한다.
5. 최종적으로 마커를 건물 쪽이 아니라 카메라 쪽으로 당겨 배치한다.

즉 현재는 "건물 앵커 위치는 유지하되, 사용자에게 보이는 마커는 건물에서 앞으로 당겨온다"가 기본 구조다.

### 8-2. preview 핀도 같은 당김 로직을 탔다

현재 preview 핀도 텍스트 카드와 같은 `BuildingMarker` 오브젝트 아래에 있다. 따라서 `ApplyMarkerPlacement()`가 marker 전체를 이동시키므로, preview 핀도 같은 당김 로직을 그대로 탄다.

즉 지금은:

- 텍스트 카드만 당겨오는 구조가 아님
- preview 핀도 같이 카메라 쪽으로 당겨짐

## 9. 마커 상태: 핀과 텍스트는 어떻게 전환되는가

현재 `BuildingMarker`는 세 상태를 가진다.

- `Hidden`
- `Preview`
- `Selected`

상태별 의미는 다음과 같다.

- `Preview`
  - preview 핀만 보임
- `Selected`
  - 텍스트 카드만 보임
- `Hidden`
  - 둘 다 숨김

즉 현재 정책은 "선택된 건물은 핀이 아니라 텍스트 카드로 승격"이다. 선택되면 preview 핀은 사라지고 텍스트 카드만 남는다.

## 10. preview 핀 UI

예전에는 preview가 단순 구체 점이었다. 현재는 사용자가 의미를 더 쉽게 이해할 수 있도록 `핀 구조`로 바뀌었다.

현재 preview 핀은 다음 파트로 구성된다.

- `PinHead`
- `PinStem`
- `PinTip`

즉 단순 점이 아니라, 위치 표시 성격이 더 강한 작은 3D 핀이다.

### 10-1. 핀 크기 제어

현재 핀 크기 제어는 단순화되어 있다.

- 실제 핀 크기 조절 값: `previewPinSize`
- 핀 모양 비율: 내부 상수
  - `PreviewPinHeadScale`
  - `PreviewPinStemScale`
  - `PreviewPinTipScale`

`UpdatePreviewPinScale()`가 매 프레임 부모 마커 스케일을 역보정해서, 실제 핀 크기가 `previewPinSize` 한 값으로 유지되도록 맞춘다.

즉 현재는:

- 핀 크기를 키우고 싶으면 `previewPinSize`만 조절
- 파트별 스케일은 핀의 비율만 담당

## 11. 텍스트 카드 UI

선택된 건물은 `Selected` 상태가 되고, 카드형 텍스트 마커가 켜진다.

카드 구성은 다음과 같다.

- 제목
- 부제
- 거리
- 어두운 배경판 `ButtonPlate`

텍스트는 `TextMeshPro`로 생성되고, 매 프레임 카메라를 향하는 billboard 형태로 회전한다.

### 11-1. 잘려 보이던 문제와 수정 내용

이전에는 `ButtonPlate`가 텍스트보다 카메라 쪽 앞에 있어, 각도에 따라 텍스트가 배경판 뒤로 가려지는 현상이 있었다.

현재는 다음처럼 수정되어 있다.

- 배경판 `sortingOrder = 0`
- 텍스트 `sortingOrder = 1`
- preview 핀 `sortingOrder = 2`
- 배경판 위치는 텍스트 뒤쪽 `z = -0.06`
- 텍스트는 약간 앞쪽 `z = 0.02`

즉 현재 구조는 예전처럼 "배경판이 글자를 덮는" 쪽이 아니라, 텍스트가 배경판 앞에서 안정적으로 보이도록 바뀌었다.

## 12. 상세보기 버튼이 켜지는 조건

선택된 건물이 있다고 해서 항상 상세보기 버튼이 활성화되는 것은 아니다. 현재는 `IsSelectedMarkerVisibleOnScreen()`가 true여야 한다.

기준은 다음과 같다.

- `Selected` 텍스트가 실제 켜져 있어야 함
- `WorldToViewportPoint()` 결과 `z > 0`
- 좌우/상하 margin 안쪽에 들어와야 함

즉 선택은 됐더라도, 텍스트 카드가 화면 경계 밖이면 상세보기 버튼은 꺼질 수 있다.

## 13. 현재 사용자 경험을 한 줄로 정리하면

현재 코드 기준 전체 흐름은 다음 한 문장으로 정리할 수 있다.

사용자가 휴대폰으로 주변을 보면, 앱은 현재 위치와 방향을 기준으로 보일 수 있는 건물 후보를 만들고, 그중 넓은 범위의 건물은 preview 핀으로 유지하며, 중심에 가까운 1개 건물만 텍스트 카드로 승격해 AR 공간에 표시한다.

## 14. 현재 구조의 핵심 특징

- 건물 인식은 이미지 인식이 아니라 위치/방향 기반 후보 선택이다.
- 건물 좌표는 건물 전체 중심이 아니라 대표 POI 좌표다.
- 앵커 고도는 현재 휴대폰 카메라 고도 기준이다.
- preview 핀은 거리 때문에 먼저 사라지지 않도록 유지된다.
- 텍스트 선택은 여전히 `detectionRadius`와 중심 판정 기준을 따른다.
- preview 핀과 텍스트 카드 모두 건물에서 카메라 쪽으로 당겨 배치된다.
- preview 핀은 단순 점이 아니라 3D 핀 UI다.
- 핀 크기는 `previewPinSize` 한 값으로 조절한다.
- 텍스트 카드의 배경판/텍스트 깊이 충돌 문제는 현재 수정된 상태다.

## 15. 현재 한계

현재 구조에도 여전히 한계는 남아 있다.

- 건물 전체 실루엣을 보는 것이 아니라 대표 좌표 1점 기준이다.
- 실제 카메라 영상의 정확한 3D 가림 정보는 사용하지 않는다.
- 세로 방향 판정은 정교한 `viewportY` 계산이 아니라 pitch 제한 위주다.
- 가로모드는 반경/각도/개수는 넓어졌지만, 중앙 텍스트 1개라는 기본 정책은 유지된다.

즉 현재 구현은 "현실 공간 위에 정확한 건물 메쉬를 붙이는 시스템"이 아니라, "위치 기반으로 가장 타당한 건물을 선택해 AR 마커로 보여주는 시스템"으로 이해하는 것이 맞다.
