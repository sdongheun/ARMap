# 프로젝트 시나리오

## 목적

이 문서는 현재 프로젝트 코드 기준으로, 사용자가 앱을 실행한 뒤 휴대폰 카메라로 실제 건물을 바라보며 정보를 확인하는 과정이 개발자 관점에서 어떻게 처리되는지 단계별로 정리한 문서다.

핵심 전제는 다음과 같다.

- 이 앱은 카메라 영상에서 건물의 외곽선을 직접 비전 모델로 분할하거나 OCR로 건물명을 읽는 구조가 아니다.
- 현재 구현은 `GPS + ARCore Geospatial Pose + 기기 Heading + 화면 중심 판정 + Kakao 장소 데이터`를 조합해 "사용자가 지금 어느 건물을 보고 있는가"를 추론한다.
- 즉 "건물 인식"은 이미지 기반 객체 인식이라기보다, 위치 기반 건물 후보 집합 중에서 사용자의 시선 방향과 가장 일치하는 대상을 선택하는 방식이다.

관련 핵심 스크립트는 아래와 같다.

- `ARmap_Pro_Max_Ultra_Plus/Assets/Scripts/Geospatial/GeospatialManager.cs`
- `ARmap_Pro_Max_Ultra_Plus/Assets/Scripts/Geospatial/GeospatialManager.Search.Flow.cs`
- `ARmap_Pro_Max_Ultra_Plus/Assets/Scripts/Geospatial/GeospatialManager.Clustering.Flow.cs`
- `ARmap_Pro_Max_Ultra_Plus/Assets/Scripts/Geospatial/GeospatialManager.Buildings.Factory.cs`
- `ARmap_Pro_Max_Ultra_Plus/Assets/Scripts/AnchorVisibilityPlanner.cs`
- `ARmap_Pro_Max_Ultra_Plus/Assets/Scripts/AnchorPoolPlanner.cs`
- `ARmap_Pro_Max_Ultra_Plus/Assets/Scripts/BuildingMarker.cs`
- `ARmap_Pro_Max_Ultra_Plus/Assets/Scripts/ARUI/ARUIManager.Cards.cs`
- `ARmap_Pro_Max_Ultra_Plus/Assets/Scripts/ARUI/ARUIManager.Detail.cs`

## 1. 앱 실행 직후: 위치와 AR 트래킹 초기화

사용자가 앱을 켜면 `GeospatialManager.Start()`가 먼저 실행된다.

1. `LocalApiKeys.json`에서 Kakao API 키와 TMap 키를 읽는다.
2. `Camera.main.transform`을 캐시한다.
3. `ARUIManager` 이벤트를 연결한다.
4. 위치 권한을 확인한 뒤 `Input.location.Start()`로 GPS를 시작한다.
5. 위치 서비스가 `Running` 상태가 될 때까지 대기한다.
6. 현재 GPS 좌표 기준으로 `ReloadDataAtCurrentLocation()`를 호출해 주변 건물 데이터를 불러온다.
7. 이후 `EarthManager.EarthTrackingState == TrackingState.Tracking`이 될 때까지 기다린다.

이 시점의 사용자 경험은 다음과 같다.

- 위치 권한이 없으면 "위치 권한이 필요합니다." 토스트를 띄우고 진행이 중단된다.
- 위치는 살아 있지만 Geospatial Tracking이 아직 안정화되지 않으면 상단 상태 배지에 `트래킹 안정화 중`이 표시된다.

## 2. 주변 장소 수집: 건물의 원재료 데이터 확보

건물 정보의 출발점은 카카오 로컬 API다. `FetchAndClusterNearbyPlaces()`에서 현재 위도/경도를 기준으로 주변 장소를 여러 방식으로 수집한다.

### 2-1. 어떤 검색을 하는가

현재 구현은 세 가지 검색을 합쳐서 후보를 만든다.

- 상권 카테고리 검색
  - 예: 마트, 편의점, 카페, 병원, 약국, 음식점, 학교
- 캠퍼스 관련 카테고리 검색
- 키워드 검색
  - 예: `인제대학교`, `인제대학교 본관`, `인제대학교 도서관`, `인제대 원룸`, `인제대 술집`

기본 반경은 `searchRadius = 200m`, 페이지는 `maxSearchPages = 3`까지 순회한다.

### 2-2. 중복 제거는 어떻게 하는가

검색 결과는 `MergeFetchedPlaces()`에서 병합된다.

- 동일 장소 판단 키는 `정규화된 장소명 + 주소` 우선이다.
- 주소가 없으면 `정규화된 장소명 + 좌표(소수점 5자리)`를 사용한다.
- 같은 장소가 여러 검색에서 중복으로 나오면 품질 점수가 높은 문서를 남긴다.

품질 점수는 대략 다음 요소로 올라간다.

- 어떤 검색 소스에서 왔는가
- 도로명 주소가 있는가
- 전화번호가 있는가
- 카테고리 그룹 정보가 있는가

즉, 앱은 단순히 API 응답을 그대로 쓰지 않고, 여러 검색 결과를 하나의 더 신뢰도 높은 장소 목록으로 정리한 뒤 다음 단계로 넘긴다.

## 3. 건물화 단계: 여러 POI를 하나의 BuildingData로 묶기

사용자가 실제로 보게 되는 대상은 개별 상점 문서가 아니라 `BuildingData`다. 이 객체가 "하나의 건물"을 대표한다.

### 3-1. 주소 그룹화

`ClusterAndBuildList()`는 먼저 결과를 `GetBestAddress(document)` 기준으로 그룹화한다.

- 도로명 주소가 있으면 도로명 주소 사용
- 없으면 지번 주소 사용

즉, 같은 주소를 공유하는 장소들은 일단 같은 건물 후보군으로 본다.

### 3-2. 실제 클러스터 분리

주소가 같다고 항상 같은 건물은 아니므로 한 번 더 나눈다.

- 일반 건물은 `clusterRadius = 20m` 기준 거리 클러스터링
- 학교, 병원, 관, 동 같은 기관형 건물은 별도 대표 토큰 기반 클러스터링

기관형 건물의 경우 다음 과정을 거친다.

1. 대표 건물처럼 보이는 문서를 고른다.
2. `본관`, `도서관`, `학생회관`, `연구동` 같은 토큰으로 건물 그룹 키를 만든다.
3. 같은 토큰을 가진 시설들을 그 대표 건물 클러스터에 붙인다.
4. 명시적 토큰이 없는 문서는 대표 건물과의 거리로 가장 가까운 클러스터에 배정한다.
5. 어느 곳에도 못 붙는 문서는 다시 거리 기반 클러스터링으로 처리한다.

### 3-3. BuildingData는 어떻게 완성되는가

`ProcessClusterIntoBuildingData()`가 최종 `BuildingData`를 만든다.

- `buildingName`
  - 건물 대표 명칭
- `latitude`, `longitude`
  - 앵커를 생성할 대표 좌표
- `description`
  - 건물 카테고리 또는 대표 설명
- `lotNumberAddress`, `roadAddress`, `fetchedAddress`
  - 주소 계층
- `phoneNumber`, `placeUrl`
  - 대표 연락/링크
- `facilities`
  - 입점 시설 목록

중요한 점은 건물 좌표도 임의 평균이 아니라, `SelectAnchorRepresentativeDocument()`로 고른 대표 문서 좌표를 그대로 사용한다는 점이다. 따라서 앵커 위치는 "건물을 가장 잘 대표하는 POI"의 좌표에 붙는다.

## 4. 앵커 후보 선정: 모든 건물에 바로 앵커를 만들지 않는 이유

앱은 주변 모든 건물에 AR Anchor를 한 번에 생성하지 않는다. 먼저 생성 가능한 건물 후보를 추린다.

`CreateAllBuildingAnchors()` 흐름은 다음과 같다.

1. 기존 앵커와 마커를 정리한다.
2. `GetBuildingsForAnchorCreation()`으로 현재 위치 기준 건물을 다시 걸러낸다.
3. `anchorCreationRadius = 150m` 안쪽 건물만 유지한다.
4. 가까운 순으로 정렬한 뒤 최대 `maxWorldTextMarkers = 20`개까지만 앵커 후보로 둔다.
5. Geospatial Tracking이 안정될 때까지 `WaitForTrackingBeforeAnchors()`로 대기한다.
6. 각 건물 altitude를 `카메라 현재 altitude + markerAltitudeOffsetMeters`로 임시 보정한다.
7. 이 단계에서는 바로 모든 앵커를 생성하지 않고, "후보 목록"만 준비한다.

여기서 중요한 설계 포인트는 다음이다.

- 데이터 로드 단계에서 건물 목록을 준비한다.
- 실제 앵커 생성은 사용자의 현재 시야 후보에 맞춰 동적으로 최소 집합만 유지한다.
- 즉, 데이터 풀과 앵커 풀을 분리해서 관리한다.

## 5. 사용자가 건물을 바라볼 때: 실제 "건물 인식"이 일어나는 방식

이 앱의 핵심은 `Update()`에서 매 프레임 실행되는 `CheckBuildingDetection()`이다.

### 5-1. 입력값

매 프레임 아래 값들이 사용된다.

- 현재 Geospatial Pose
  - `EarthManager.CameraGeospatialPose`
- 현재 AR Tracking 상태
  - `EarthManager.EarthTrackingState`
- 사용자의 현재 방향
  - `pose.Heading`
- 카메라 전방 벡터
  - `_cameraTransform.forward`
- 후보 건물 목록
  - `_anchorCandidateBuildings`

### 5-2. 1차 후보: 시야각 안에 들어오는 건물

`AnchorVisibilityPlanner.CalculateVisibleCandidates()`가 먼저 "보일 수 있는 건물"을 찾는다.

기준은 다음과 같다.

- Earth Tracking이 유효해야 한다.
- Heading Accuracy가 임계값보다 좋아야 한다.
- 건물까지 거리가 `anchorPreviewRadius` 이하여야 한다.
- 현재 사용자의 Heading과 건물 방위각의 차이가 `detectionAngle / 2` 안에 있어야 한다.

기본 설정은 대략 다음과 같다.

- `anchorPreviewRadius = 250m`
- `detectionAngle = 130도`
- `detectionRadius = 100m`
- `verticalAngleLimit = 60도`

이 단계에서 각 건물은 단순 거리뿐 아니라 "현재 사용자가 고개를 어느 방향으로 두고 있는가" 기준으로 걸러진다.

### 5-3. 화면상의 가상 X 좌표 계산

이 프로젝트는 건물의 실제 3D 메쉬 투영으로 가시성을 계산하지 않는다. 대신 Heading 차이를 정규화해서 화면의 가상 `viewportX`를 만든다.

- 정면이면 `viewportX = 0.5`
- 왼쪽이면 0.5보다 작아짐
- 오른쪽이면 0.5보다 커짐

즉, "이 건물이 화면 중앙에 얼마나 가깝게 있다고 볼 수 있는가"를 방위각 차이로 근사한다.

### 5-4. 후면 후보 제거

전방 후보가 여러 개 잡히더라도 같은 시야선에 뒤쪽 건물이 겹칠 수 있다. 이를 `GetFrontMostCandidates()`가 정리한다.

- 두 건물의 viewport 위치가 충분히 가깝고
- 한 건물이 다른 건물보다 `groupedCandidateDistanceBias = 6m` 이상 멀면
- 뒤쪽 건물을 제거한다

즉, 사용자가 앞건물 뒤의 먼 건물을 같이 바라보는 상황에서도, 먼저 보이는 건물만 남긴다.

### 5-5. 최종 선택 건물 계산

`DetermineFocusedBuilding()`은 남은 후보 중에서 실제로 "지금 보고 있다"고 판단할 건물을 선택한다.

판단 요소는 다음과 같다.

- 화면 중심 `(0.5, 0.5)`과의 거리
- 건물까지의 실제 거리
- 카메라 pitch가 너무 위/아래를 향하면 선택하지 않음
- `centerViewportThreshold` 안에 들면 우선 선택
- 안 들더라도 약간 완화된 threshold 안에 있으면 fallback 선택

결과적으로 사용자가 휴대폰으로 특정 건물을 정면에 가깝게 두고 바라보면 그 건물이 `focusedBuilding`이 된다.

## 6. 앵커 생성: 선택 후보만 실제 ARGeospatialAnchor로 올리기

시야 후보가 정해지면 `RequestAnchorPoolForCandidates()`가 호출된다.

### 6-1. 앵커 풀 개념

앱은 모든 건물에 영구 앵커를 두지 않는다. 현재 상위 후보 건물 집합만 앵커로 유지한다.

1. `topCandidates`에서 중복 없는 `desiredBuildings`를 만든다.
2. 이전 프레임과 동일한 집합이면 아무것도 하지 않는다.
3. 달라졌으면 `SyncAnchorPool()`을 실행한다.

`SyncAnchorPool()`은 다음과 같이 동작한다.

- 더 이상 필요 없는 앵커는 제거
- 새로 필요한 건물만 순차적으로 생성
- 유지할 필요가 있는 기존 앵커는 재사용

이 구조 덕분에 사용자가 회전하거나 이동할 때, 현재 보고 있는 방향에 필요한 건물만 앵커가 살아 있게 된다.

### 6-2. 실제 앵커 생성

`CreateAnchorForBuilding()`에서 실제 앵커를 만든다.

```csharp
ARGeospatialAnchor anchor = AnchorManager.AddAnchor(
    building.latitude,
    building.longitude,
    building.altitude,
    Quaternion.identity);
```

이 순간 생성되는 것은 월드 좌표가 아니라 지구 좌표계 기준 앵커다.

- 위도
- 경도
- 고도

ARCore Extensions가 이 지구 좌표를 현재 AR 세션 월드 공간으로 매핑한다. 따라서 사용자가 위치를 조금 바꾸거나 회전해도, 트래킹이 안정적인 동안 마커는 실제 공간의 동일 위치에 고정된 것처럼 보인다.

## 7. 텍스트 마커 생성: 앵커 위에 어떤 오브젝트가 붙는가

앵커가 생성되면 그 자식으로 `buildingMarkerPrefab`이 인스턴스화된다.

```csharp
GameObject markerObj = Instantiate(buildingMarkerPrefab, anchor.transform);
```

그 뒤 `BuildingMarker`가 바인딩된다.

- `marker.BindBuilding(building)`
- `marker.SetInfoContent(title, subtitle, distance)`
- `marker.SetInfoVisible(true)`
- `marker.SetState(Preview)`

### 7-1. BuildingMarker 내부 구조

`BuildingMarker`는 런타임에 다음 요소를 만든다.

- `TextMarker` 루트
- 미리보기용 구(Sphere) 점
- 배경 판(Quad)
- 제목 텍스트
- 서브타이틀 텍스트
- 거리 텍스트
- 터치용 BoxCollider

즉, 사용자가 화면에서 보는 "텍스트 마커"는 단일 텍스처가 아니라 3D 월드 공간 안에 배치된 작은 라벨 UI 오브젝트다.

### 7-2. 마커 상태

마커는 크게 세 상태를 가진다.

- `Hidden`
  - 아무것도 보이지 않음
- `Preview`
  - 작은 점만 보임
- `Selected`
  - 텍스트 패널이 열림

현재 구현에서 중요한 부분은 다음이다.

- 후보 건물이지만 현재 중앙 선택 대상이 아니면 `Preview`
- 중앙에서 실제로 바라보는 건물로 판정되면 `Selected`
- 시야 후보가 아니면 `Hidden`

즉, 사용자는 먼저 점 형태의 후보를 보고, 그중 화면 중심에 들어온 건물만 큰 텍스트 라벨로 확장된다고 이해하면 된다.

## 8. 사용자 위치에서 마커가 어떻게 보이는가

### 8-1. 월드에 고정된 위치

마커의 부모는 `ARGeospatialAnchor`다. 따라서 마커는 카메라 앞 2D HUD에 붙는 것이 아니라 실제 공간상의 건물 방향에 고정된다.

사용자 관점에서는 다음과 같이 보인다.

- 내가 고개를 돌리면 마커가 함께 이동하지 않는다.
- 건물 방향을 향할 때만 그 방향에 라벨이 남아 있다.
- 건물에서 시선을 떼면 라벨이 화면 바깥으로 사라지거나 다른 후보로 교체된다.

### 8-2. 라벨은 항상 카메라를 향함

`BuildingMarker.Update()`에서 아래 처리를 한다.

- `_visualRoot.forward = _visualRoot.position - _cachedCameraTransform.position`

즉 라벨 평면은 항상 카메라를 정면으로 바라보는 빌보드 방식이다. 사용자가 옆으로 이동해도 텍스트가 돌아가며 정면으로 보이기 때문에 읽기 쉬워진다.

### 8-3. Preview와 Selected의 시각 차이

- `Preview`
  - 점만 활성화
  - 먼 건물 후보를 간단히 암시
- `Selected`
  - 제목, 설명, 거리 라벨이 들어간 패널 활성화
  - collider 활성화
  - 터치 가능

따라서 사용자는 한 번에 너무 많은 텍스트를 보는 것이 아니라, 현재 바라보는 핵심 건물에만 정보 패널이 열린다.

### 8-4. 화면 안에 정말 보이는지 최종 확인

선택된 마커에 대해서는 `IsSelectedMarkerVisibleOnScreen()`이 다시 확인한다.

- `WorldToViewportPoint()`로 실제 화면 투영
- `z > 0`인지 확인
- 좌우/상하 margin 안에 들어오는지 확인

이 값이 참일 때만 상세 정보 버튼도 활성화된다. 즉, 논리적으로 선택되었더라도 실제 화면에 잘 보이지 않으면 UI 액션을 제한한다.

## 9. 사용자가 보는 전체 UX 흐름

### 단계 A. 앱을 켠 직후

- 위치/GPS 시작
- Geospatial Tracking 안정화 대기
- 상단 배지에 `트래킹 안정화 중`

### 단계 B. 주변 데이터 확보

- Kakao API에서 주변 장소를 수집
- 중복 제거
- 주소/거리/기관형 규칙으로 건물 단위로 묶음
- 상단 배지에 `건물 N개 탐지됨`

### 단계 C. 아직 특정 건물을 정면으로 보지 않을 때

- 시야각 안의 후보 건물은 계산됨
- 하지만 화면 중심 건물이 아니면 `focusedBuilding = null`
- 상단 배지에 `건물을 바라보세요`
- 월드에는 preview 점 마커만 보일 수 있음

### 단계 D. 사용자가 특정 건물을 정면으로 볼 때

- Heading 기준으로 해당 건물이 중심 후보가 됨
- 같은 시야선의 뒤쪽 후보는 제거됨
- 그 건물의 앵커가 풀에 포함됨
- `BuildingMarker`가 `Selected` 상태로 바뀜
- 건물명, 카테고리, 거리 텍스트가 월드 공간에 표시됨
- 상단 배지는 숨겨짐

### 단계 E. 사용자가 마커를 터치할 때

- `Physics.Raycast`로 마커 collider를 맞춘다
- 바인딩된 `BuildingData`를 꺼낸다
- `ARUIManager.OpenDetailView()` 호출
- 상세 패널에서 주소, 전화, 시설 목록, 지도 링크 등을 보여준다

## 10. 개발자 관점의 핵심 해석

이 프로젝트에서 "건물을 인식한다"는 말은 현재 코드 기준으로 아래 의미다.

1. GPS 주변 반경에서 건물 후보 데이터를 미리 수집한다.
2. 그 후보를 건물 단위 객체로 재구성한다.
3. 사용자의 현재 위치와 방위각으로 볼 수 있는 건물만 추린다.
4. 화면 중심에 가장 가까운 건물을 현재 바라보는 건물로 선택한다.
5. 그 건물의 지리 좌표에 Geospatial Anchor를 생성한다.
6. 그 앵커 자식으로 텍스트 마커를 붙인다.
7. 마커는 월드에 고정되되, 텍스트 면은 카메라를 향하게 해서 읽히도록 한다.

즉, "카메라가 건물을 알아보는 것"이 아니라 "앱이 현재 위치와 시선 방향으로 사용자가 어떤 건물을 볼 가능성이 가장 높은지 계산하고, 그 건물 좌표에 AR 콘텐츠를 정렬하는 것"이 정확한 설명이다.

## 11. 현재 구현 기준의 한계

현재 구조의 특성상 아래 한계가 있다.

- 실제 영상 기반 건물 외곽 인식이 아니므로, GPS 오차와 Heading 오차에 민감하다.
- 건물 정면/후면, 가림막, 층수 정보는 정교하게 반영되지 않는다.
- 고도는 현재 카메라 altitude에 오프셋을 더하는 fallback 방식이라, 건물별 정확한 지면/입면 고도와 다를 수 있다.
- 같은 방향선에 여러 건물이 있으면 거리 기반으로 뒤 후보를 제거하지만, 실제 occlusion 계산까지 하지는 않는다.

그럼에도 현재 구조는 다음 목적에는 적합하다.

- 캠퍼스/상권 주변 건물 탐색
- 사용자가 지금 보고 있는 건물의 이름/시설 정보 오버레이
- 빠른 프로토타이핑 및 실시간 AR 안내

## 12. 한 줄 요약

현재 앱은 "카메라 영상을 직접 분석해서 건물을 식별하는 시스템"이 아니라, "현재 위치에서 수집한 건물 데이터와 사용자의 시선 방향을 결합해 가장 가능성 높은 건물을 선택하고, 그 좌표에 Geospatial Anchor와 텍스트 마커를 띄우는 시스템"이다.
